# Network-Packet-Sniffer-Analyzer
A lightweight CLI tool that captures raw network packets, decodes them down to the protocol level (Ethernet -> IPv4 -> TCP/UDP), and flags TCP traffic that matches known stealth-scan signatures. Built in Python first, transitioned to C later

Architecture

## Architecture
 
```
   NIC / kernel
        |
        v
 +-------------+      bounded queue       +------------------+

 |  thread)    |                          +------------------+
 +-------------+                                   |
                                                    v
                                            +---------------+

                                            | Eth/IP/TCP/UDP |   Ethernet->IP->TCP/UDP)
```

### Design decisions
 
**Producer-consumer / Bounded Queue.** The capture thread (`capture.py`) does nothing but `recv()` and enqueue (zero parsing & zero analysis). This means a slow detector can never delay the next packet read. The queue is bounded and uses a "drop-oldest" policy when full: under an active scan, the most recent traffic is what is actually detected, so dropping stale queued packets is the right tradeoff over either blocking the producer or letting memory grow unbounded.
 
**Parsing layer.** `parser.py` takes raw bytes in, returns a
`PacketInfo` out, with no memory between calls. Mirrors the way the kernel network stack itself demultiplexes a frame (EtherType field decides IPv4 vs. ARP vs. IPv6; IP protocol field decides TCP vs. UDP); it's pure CPU-bound struct unpacking with no I/O or locking to complicate a rewrite (good for transition to C).
 
**Strategy pattern for detectors.** Every detector implements
`inspect(packet) -> Optional[Alert]`. The engine doesn't know or care how any individual detector works internally. Adding the next detector (port scan, traffic spike) means writing one new class (nothing else in the system changes).
 
**Alert Delivery.** The analysis engine publishes `Alert` objects to an `AlertPublisher`; it has no idea whether a `ConsoleAlerter`, a future file logger, or a future dashboard is listening.
 
**Sentinel Shutdown.** Rather than risk the consumer thread blocking forever on an empty queue, `capture.stop()` pushes a `POISON_PILL` sentinel that the consumer recognizes and exits cleanly on.
 
## Current detector: stealth-scan signatures
 
`StealthScanDetector` is intentionally stateless: it judges each TCP packet purely on its own flag byte, with no tracking across packets. That made it the cheapest possible way to validate the *entire* pipeline end-to-end (capture → parse → analyze → alert) before building anything stateful on top. It currently flags:
 
- **NULL scan** — no TCP flags set at all
- **FIN scan** — only the FIN flag set
- **XMAS scan** — FIN + PSH + URG set together
- **SYN+FIN** — mutually contradictory flags set simultaneously
## Running it
 
Requires Linux (uses `AF_PACKET` raw sockets) and root (raw sockets need
`CAP_NET_RAW`):
 
```bash
sudo python3 -m packet_sniffer.main --iface lo
sudo python3 -m packet_sniffer.main --iface eth0 --duration 30
```
 
`--iface lo` is useful for local testing; point it at a real interface
(`eth0`, `wlan0`, etc.) to monitor actual network traffic.
 
## Testing
 
Two layers of tests, deliberately:
 
```bash
python3 packet_sniffer/tests/test_parser.py   # synthetic byte buffers, no sockets
python3 packet_sniffer/tests/test_e2e.py       # real raw-socket capture on loopback
```
 
`test_parser.py` checks the decoder against hand-crafted packets with known
field values — fast, deterministic, no privileges needed. `test_e2e.py`
actually starts the live pipeline on `lo`, sends a real crafted XMAS-scan
packet over the loopback interface using a hand-built IP/TCP header (with
correct checksums), and confirms the detector fires on genuine
kernel-delivered traffic, not just hand-fed bytes.
 
Note: loopback traffic on Linux is often visible twice in capture (TX and
RX paths both get delivered to `AF_PACKET` listeners), so don't be alarmed
if packet counts look doubled when testing against `lo` specifically — this
doesn't happen on real interfaces.
